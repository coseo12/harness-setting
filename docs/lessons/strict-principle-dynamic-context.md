# 엄격 원칙 + 동적 적응 부재 = 실사용 실패

> **요지**: 단일 축 엄격 원칙 (fact-first / 1:1 정확 등) + 뷰포트·해상도 등 동적 문맥 적응 부재 = 자동 검증 PASS / 실사용 실패. CLAUDE.md 실전 교훈 "빌드 성공 ≠ 동작하는 앱" 의 원칙·UX 버전 확장.
>
> **근거**: volt [#68](https://github.com/coseo12/volt/issues/68) (astro-simulator P10~P12, 2026-04) — 사실 비율 원칙 + 동적 스케일 부재 결합 함정 실측.

---

## 핵심 명제

> **단일 축으로 강하게 선언된 원칙 + 동적 문맥 적응 부재 = 자동 검증은 PASS, 실사용은 실패** 라는 구조적 함정이 있다. 원칙은 "사실 / 정확 / 무결" 같은 축에서 선언되기 쉬우나, 실제 UX 는 뷰포트·해상도·카메라 거리 등 **동적 문맥** 과 상호작용한다.

원칙 박제 직후 반드시 "이 원칙이 실 뷰포트/디바이스 분포에서 어떻게 작동하는가" 시뮬레이션을 거친다. 자동 검증(canvas pixel readback / console error 0) 이 PASS 라도 실사용 실패는 별개 종류의 리스크다.

## 관찰 맥락 (astro-simulator P10~P12, 2026-04)

- P10 에서 `docs/principles/fact-first.md` 박제: "사실(fact)은 1차, 시각은 2차 overlay"
- `educational` (과장 배수 ×20/×500) / `scientific` (1:1 실측) 이중 모드 도입
- scientific 모드 float32 jitter 해소를 위해 P11-A Floating Origin 까지 구현·머지 완료
- DoD 4 실 Chrome 육안 검증 중 사용자 관찰:

  > "focus 해도 행성이 안 보여 그리고 궤도에서 움직임도 보여야하는데 근본적으로 스케일에 대한 토론이 필요해"

- 자동 browser-verify (canvas pixel readback / console error) 는 **전부 PASS** 였다

### 구조적 원인

| 축 | 원칙 | 결과 |
|---|---|---|
| 사실성 | 상대 비율 실측 고정 | body 간 비율은 IAU 2015 기준 정확 |
| 시각성 | 동적 스케일 부재 | 고정 ×1 (scientific) → sub-pixel / 고정 ×20/×500 (educational) → 임의 비율 왜곡 |

두 축 중 어느 하나를 지키면 다른 하나가 깨지는 구조. 태양:지구 = 109:1 → 태양 64px 일 때 지구 0.59px = sub-pixel 실사용 실패 확정.

### 해결 원칙 (재설계 합의)

사용자가 제시한 5원칙:

1. 상대 비율은 실측 고정 (왜곡 금지)
2. **절대 스케일은 디스플레이 함수** — 뷰포트/zoom 에 따른 공통 동적 배수
3. 모드 통일 (이중 모드 분리 자체 폐기)
4. 거리도 동일 스케일 (궤도 반지름 포함)
5. 화면 이동은 자연스러워야 함

핵심은 **"뷰포트의 함수로 스케일을 정의"** 하는 관점 전환. 고정 상수 대신 tier 기반 동적 renderScale. 모든 body 는 동일 배수로 확대되므로 실측 비율이 자동 보존되면서 가시성도 확보.

## 사전 감지 체크리스트 (유사 도메인)

과학 시각화 / AR·VR / 교육 / GIS / 분자 시뮬 / 천체 시뮬 등 **극단적 스케일 차이** 가 있는 제품 초기 설계 시:

- [ ] "실측 비율을 그대로 렌더링" 요구가 **뷰포트 해상도** 와 함께 명시되는가? ("1280×800 에서 N px" 류)
- [ ] "최소 가시 크기" (min-visible-px floor) 혹은 **뷰포트 기반 동적 스케일** 전략이 설계에 있는가?
- [ ] 이중 모드 분리 (sim/edu) 가 아닌 **단일 동적 모드** 로 통일 가능한지 검토했는가?
- [ ] DoD 에 **"화면에 의미 있는 무언가가 보인다"** 항목이 수치로 포함됐는가? (예: focus 대상 ≥ N px)
- [ ] 브라우저 자동 검증이 "canvas 가 검정이 아님" 이상의 **도메인 특화 pixel 검증** 을 포함하는가?

## 일반화된 두 명제

1. **엄격 원칙 + 동적 적응 부재 = 실사용 실패.** 원칙은 단일 축에서 강하게 선언되기 쉬우나 실 UX 는 동적 문맥과 상호작용. 박제 직후 실 뷰포트 시뮬레이션 의무.
2. **완료된 Phase 의 성공 기준이 실사용에서 깨지는 현상은 자동 검증으로 감지 불가.** 사용자 실 디바이스 육안 검증을 DoD 의 일급 시민으로 포함. CRITICAL #3 (UI 3단계 브라우저 검증) 의 원칙·UX 버전.
