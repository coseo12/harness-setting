# ADR 작성

코어 기술 결정은 미래의 자기/팀/에이전트가 "왜 이걸 골랐지"를 재구성할 수 있도록 ADR로 박제한다.

## 사전 조건

```bash
# 디렉토리 확인 — 없으면 생성
mkdir -p docs/decisions
```

## 파일명 규약

`docs/decisions/<YYYYMMDD>-<kebab-case-topic>.md`

예: `20260415-wasm-language-choice.md`, `20260420-state-management-zustand.md`

## 본문 구조 (헤더명 고정)

```markdown
# ADR: {결정 제목}

- **상태**: Proposed | Accepted | Superseded by [#…] | Deprecated
- **날짜**: YYYY-MM-DD
- **결정자**: {이름/역할}

## 배경

어떤 문제·제약·요구사항이 있어서 결정이 필요해졌는가. 1~3 단락.

## 후보 비교

| 후보 | 장점 | 단점 | 비고 |
|---|---|---|---|
| A | … | … | … |
| B | … | … | … |

비교 축은 결정에 실제로 영향을 준 것만 나열 (성능/생태계/러닝커브/유지비/팀 역량 등).

## 결정

선택한 것 + 핵심 근거 1~3개. 모호한 표현 금지 — 구체적 수치/시나리오로.

## 결과·재검토 조건

- 기대 효과 (측정 가능한 형태)
- 트레이드오프로 받아들인 비용
- **재검토 트리거**: 어떤 조건이 발생하면 이 결정을 다시 본다 (예: "벤치마크에서 N=1000 60fps 미달", "유지보수자 0명 6개월")

## 재도입 트리거 (폐기/보류 결정 시 필수)

상태가 `Deprecated` / `Superseded` 또는 "X 를 현 시점에 채택하지 않음" 류 폐기·보류 결정일 때 **필수**. Accepted 결정에서는 선택.

- **재도입 검토 조건**: 어떤 외부 변화가 발생하면 이 결정을 되돌려 X 를 재도입할지 — 기술 성숙도 / 사용자 전환 / 외부 표준 / 벤치마크 임계 등 **시간 함수** 로 표현
  - 예: "WebGPU 모바일 호환성 iOS Safari/Android Chrome 안정 버전 도달" / "사용자 전환율 30% 이상이 모바일" / "외부 표준 W3C Candidate Recommendation 승격"
- **Graceful Degradation 경로**: 재도입 전까지 폐기 대상 사용자가 만나는 UX (예: "모바일 접속 시 안내 모달 + 데스크톱 권장")
- **재도입 시 선행 작업**: 재도입을 결정했을 때 필요한 구체 단계 (테스트 매트릭스 복원 / LOD 도입 / 별도 tier 전략 등)

## 참고

- 관련 이슈/PR/외부 문서 링크
```

## 절차

1. **ADR 필요 여부 판정** — 일회성 코드 결정이면 ADR 불필요. 다음 중 하나면 작성:
   - 코어 언어/런타임/프레임워크 도입·교체
   - 주요 외부 의존성 추가 (DB, 메시지 큐, 인증 등)
   - 프로젝트 전반에 영향을 주는 패턴 채택 (상태 관리, 빌드 도구, 모노레포 구조)
2. **기존 ADR 검색** — `ls docs/decisions/` + 키워드 grep으로 중복/대체 대상 확인. 대체 시 기존 ADR 상태를 `Superseded by [신규 파일명]`으로 갱신.
3. **드래프트 작성** — 위 템플릿. 후보 비교는 실제로 검토한 것만 (가짜 후보 금지).
4. **사용자 검토** — 결정 섹션과 재검토 조건은 사용자와 합의 후 확정.
5. **커밋** — `docs(adr): {topic} 결정 기록` 형식. 보통 결정을 만든 PR과 함께 커밋.

## Concrete Prediction (volt #47)

ADR 결정을 구현할 때 **"코드 변경 0 예측"** 을 박제하면 기존 추상화의 건강성을 실증할 수 있다. 레이어/플러그인/스키마 확장 결정에 특히 유용. (아래 "NO-OP ADR 변형" 과 구분 필수 — NO-OP 는 "이미 해소됐으므로 구현 불필요", Concrete Prediction 은 "구현하되 코드 변경 0 으로 가능". 산출물 유/무가 핵심 차이.)

### 예측 박제 형식

ADR 본문 `## 결과·재검토 조건` 섹션에 다음 형태로 추가:

```markdown
### Concrete Prediction

- **{신규 요소} 추가 시**: {핵심 모듈 경로} 코드 변경 **0 줄** (데이터 추가만)
  - 검증 방법: `git diff --stat <경로>` — 실구현 PR 에서 경로 아래 `*.ts`/`*.js` 변경 0건이면 예측 성공
  - 실패 시 대응: 계층에 데이터 참조가 아닌 하드코딩 잔존 → (a) 먼저 리팩토링 후 구현 재개 (b) 예외 인정 후 ADR Amendment 박제
```

### 적용 조건

- 데이터(JSON/YAML/DB row) 로 확장 가능한 **계층적 구조** 일 때 — parentId 체인 / 플러그인 레지스트리 / 라우팅 테이블 / 스키마-주도 UI / i18n
- ADR 결정 자체가 "기존 추상화로 수용 가능" 을 전제할 때만 적용 — 새 모듈/레이어를 만드는 결정에는 적용 불가
- 예측 박제는 **reviewer/QA 가 실측할 수 있는 검증 가능 기준** 형태여야 한다 (수치·경로 명시)

## NO-OP ADR 변형 (volt #14)

인계받은 작업이 착수 시점 실측에서 **이미 해소돼 있을 때**, 구현 대신 NO-OP ADR로 종결할 수 있다. 파일명 관례: `<YYYYMMDD>-<topic>-no-op.md`.

포함 내용:
- **배경**: 어떤 인계/가정이 있었는가
- **실측 결과**: 현재 동작 (수치·로그·스크린샷 등 증거)
- **결정**: 변경 NO-OP — 왜 구현이 불필요한가
- **대신 박제**: 회귀 가드(테스트/verify 스크립트 경로) — 실측 동작이 퇴행하지 않도록
- **재검토 조건**: 어떤 상황에서 이 결정을 다시 봐야 하는가

NO-OP ADR은 "아무것도 안 했다"는 기록이 아니라 "왜 아무것도 안 해도 되는지"의 근거. 다음에 재발굴 시 빠르게 기각하는 박물관 전시물이 된다.

## 변형 박제 — D' (prime) 패턴 (volt #26)

ADR 원안을 구현하는 도중 **기술 장벽**(프레임워크 버그, 플랫폼 제약 등)으로 결정을 현실에 맞게 조정해야 할 때, 원안을 **편집하거나 폐기하지 않고** 변형 서브섹션을 추가 박제한다. reviewer·QA·후속 작업자가 차이를 인지하고 판단할 수 있도록 의사결정 이력을 보존하는 패턴.

### 박제 구조

원안 `## 결정` 섹션 **아래**에 변형 서브섹션을 덧붙인다. 원안 본문은 절대 편집하지 않는다 (이력 보존).

```markdown
## 결정

{원안 결정 — 편집 금지}

### D' 변형 ({YYYY-MM-DD})

- **변경 이유**: {기술 장벽 — 구체적 증상·재현 조건. 가능하면 volt 이슈 링크}
- **본질 영향**: {DoD 충족 여부. 스프린트 계약 기준 어디를 만족/포기하는가}
- **원안 대비 차이**: {무엇이 바뀌고 무엇이 유지되는가 — 기술적 매핑}
- **원안 복원 트리거**: {어떤 조건이 해소되면 D 원안으로 되돌릴지 — fix 이슈 번호 링크}
```

### 병행 조치

1. **후속 fix 이슈 스핀아웃** — 원안 복원을 위한 fix 이슈를 별도로 생성하고 ADR 에 역링크. fix 이슈에는 **시도한 옵션 / 실패 스크린샷·로그 / 추정 원인**을 박제해 다음 작업자의 재시작 비용을 낮춘다.
2. **PR 본문에 "⚠️ ADR 변경 명시" 섹션 필수** — 머지 후 ADR이 원안과 다른 상태임을 reviewer/후속 작업자가 즉시 인지할 수 있게 한다. 머지된 ADR을 그대로 믿으면 안 된다는 신호.
3. **사용자 승인 필수** — architect 결정 이탈이 감지되면 자동 진행 금지. 옵션 A(ADR 정정 + 진행) / B(재시도) / C(범위 축소) 를 사용자에게 제시하고 명시적 승인을 받는다.

### 변형 박제 vs 폐기 vs 재작성

| 상황 | 선택 | 이유 |
|---|---|---|
| 기술 장벽으로 원안이 현실과 어긋남 | **변형 박제 (D')** | 원안 의도 보존, 복원 가능 |
| 원안 자체가 틀렸음이 확정 | Superseded by 새 ADR | 이력 단절 없이 대체 기록 |
| 구현 범위만 축소, 의도 유지 | 변형 박제 | 범위 축소도 변형 |
| 결정 자체가 무의미해짐 | Deprecated 표기 | 폐기도 이력은 남긴다 |

"변형"은 "실패"가 아니다 — 설계 학습의 기록이자 미래 복원의 근거다.

## 금지/주의

- **사후 정당화 금지** — 이미 코드를 작성한 뒤 "결정 근거를 끼워 맞추기" 식으로 쓰지 않는다. ADR은 결정 근거를 구조화하는 도구이지 합리화 도구가 아니다.
- **후보 단일화 금지** — "다른 선택지 없음" 식 ADR은 가치가 낮다. 최소 2개 후보를 비교한다 (기각 사유 명시).
- **모호 결론 금지** — "상황에 따라" 같은 표현 금지. 결정은 단일하고 명시적이어야 한다.
- **재검토 조건 누락 금지** — 어떤 조건에서 이 결정이 무효화되는지를 적지 않으면 ADR이 박물관 전시물이 된다.
- **폐기·보류 결정의 재도입 트리거 누락 금지** (volt #55) — "영구 폐기", "지원 안 함", "무기한 중단" 프레이밍으로 끝내지 않는다. 기술 성숙도는 시간 함수이므로 "재도입 검토 조건" 을 명시하지 않으면 결정이 **죽은 부채** 로 고정된다. `Deprecated` / `Superseded` 상태 ADR 은 `## 재도입 트리거` 섹션 필수 (위 템플릿 참조). 관련 Claude 편향: CLAUDE.md `## 교차검증` 의 "폐기 프레이밍 선호" 체크리스트 항목.
